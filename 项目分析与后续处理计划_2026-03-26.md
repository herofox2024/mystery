# 推理资讯周报项目分析与后续处理计划

日期：2026-04-02

## 文档定位

本文档基于当前仓库实际状态重新整理，用于替换早期结论中已经失效的部分。

当前判断以以下事实为准：

- 当前目录是 Git 仓库
- `app_helpers.py` 已存在，且包含主流程依赖的核心函数
- 项目已经成功生成过多份 Markdown/HTML 周报
- 部分终端中的中文乱码主要是 PowerShell 输出编码问题，不等于文件本身损坏

## 项目目标

这是一个 Python 自动化周报项目，目标是：

- 抓取豆瓣推理相关新书
- 抓取 RSS 资讯与中文补充来源
- 通过规则过滤、AI 精筛和历史去重保留高相关内容
- 生成 Markdown 和 HTML 周报
- 支持将最新一期发布到 GitHub Pages，并生成历史归档页面

## 当前结构与职责

当前主流程文件：

- `main.py`：程序入口，负责加载配置、执行抓取、过滤、去重、生成报告
- `app_helpers.py`：默认配置补全、配置校验、规则过滤、打分、状态读写
- `report.py`：Markdown/HTML 周报渲染与资源处理
- `publish_pages.py`：发布最新周报到站点首页，并维护 `reports/` 历史页面
- `scrapers/`：豆瓣、RSS、中文来源、AI 筛选等采集与处理逻辑
- `config.yaml`：运行配置
- `data/state.json`：历史去重状态

## 当前主要风险

### 1. `config.yaml` 存在明文 API Key

这是当前最需要优先处理的问题。

风险：

- 密钥泄漏
- 一旦仓库同步、备份或截图外流，凭据即失控
- 与 README 中“优先使用环境变量”的约定不一致

建议：

- 立即将明文 key 移出配置文件
- 统一改为环境变量读取
- 对已经暴露的 key 进行轮换

### 2. 文档存在“旧结论覆盖新现实”的误导风险

旧版分析文档会把维护者引向错误方向，例如重复排查已不存在的问题。

风险：

- 浪费排障时间
- 错误评估项目成熟度
- 后续修复优先级失真

建议：

- 保留该文档文件名，但内容以当前状态重写
- 所有状态判断以当前代码和产物为准，不再引用旧结论

### 3. 内容质量仍依赖规则与 AI 双重约束

虽然项目可运行，但最终周报质量依然受以下因素影响：

- RSS 来源是否足够聚焦
- 关键词规则是否过宽或过窄
- AI provider 是否可用
- AI 失败时是否采用保守策略

这类问题不属于“不能运行”，而属于“结果是否足够稳定、足够聚焦”。

### 4. 当前仓库仍在变动

已观察到工作区存在未提交修改，说明当前状态不是冻结版本。

影响：

- 文档描述需要尽量基于当前代码，不做超出证据的断言
- 后续修改前应注意与现有变更协同，不应回退无关改动
